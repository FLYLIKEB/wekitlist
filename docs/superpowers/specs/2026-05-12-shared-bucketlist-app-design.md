# 공유 버킷리스트 앱 MVP 설계

## 개요
커플이나 친구가 함께 쓰는 초경량 공유 버킷리스트 앱이다.

이 제품의 목표는 두 사람이 아주 적은 마찰로 같은 공간에 들어와 함께 목표를 추가하고, 완료 처리까지 빠르게 할 수 있게 만드는 것이다. 앱은 깔끔하고 빠르고 차분해야 하며, 공유 리스트 경험에 직접 도움이 되지 않는 요소는 최대한 제거한다.

## 제품 원칙
- 매우 가벼운 PWA 스타일 웹앱
- 빠른 초기 로드와 부드러운 리스트 상호작용
- 텍스트 중심의 미니멀하고 Apple 같은 정돈된 UI
- 입력 단계를 최소화한 높은 편의성
- 불필요한 소셜 기능, 게임화, 관리 기능 배제

## 목표 경험
이 앱은 무거운 소셜 서비스보다는 공유 메모나 공유 리마인더에 가까운 느낌이어야 한다. 사용자는 앱을 열자마자 해야 할 항목을 보고, 몇 초 안에 새 항목을 추가하고, 한 번의 탭으로 완료할 수 있어야 한다.

## MVP 범위
### 포함
- 공유 리스트 1개 생성
- 초대 링크로 공유 리스트 참여
- 공유 리스트 그룹명 설정
- 버킷리스트 항목 추가
- 버킷리스트 항목 수정
- 항목 완료 처리
- 완료 항목 다시 열기
- 완료 항목 별도 보기

### 제외
- 푸시 알림
- 사용자당 여러 리스트 또는 워크스페이스
- 프로필 사진
- 리액션이나 댓글
- 인앱 채팅
- 이미지 업로드
- 전용 지도/위치 필드
- 드래그 앤 드롭 정렬
- 복잡한 리마인더
- 오프라인 우선 복잡성
- 네이티브 앱 빌드

## 핵심 정보 구조
### 공유 리스트
제품의 핵심 단위는 하나의 공유 리스트다.

UX는 기본적으로 두 사람이 함께 쓰는 상황에 최적화하지만, 내부 모델은 이후 필요 시 2인 이상도 수용할 수 있게 설계할 수 있다. 다만 MVP에서는 추가적인 그룹 관리 기능을 노출하지 않는다.

필드:
- id
- groupName
- inviteToken 또는 inviteLink
- createdAt

### 참여자
참여자 정보는 의도적으로 가볍게 유지한다.

필드:
- id
- sharedListId
- displayName
- joinedAt

프로필 이미지와 복잡한 계정 정체성은 MVP 범위에서 제외한다.

### 버킷리스트 항목
필드:
- id
- sharedListId
- title
- memo
- targetDate
- createdAt
- updatedAt
- completedAt
- createdBy
- completedBy

규칙:
- `title`은 필수다
- `memo`는 선택이다
- `targetDate` 기본값은 항목 생성일이다
- 항목을 완료하면 `completedAt`이 자동 저장된다
- 항목을 다시 열면 `completedAt`은 제거된다
- 위치 정보는 별도 필드가 아니라 `memo` 안의 일반 링크로 처리하며, 카카오맵 링크도 허용한다

## 주요 사용자 흐름
### 첫 진입
1. 사용자가 앱을 연다
2. 사용자는 공유 리스트를 만들거나 초대 링크로 입장한다
3. 사용자는 그룹명을 설정하거나 이미 설정된 그룹명을 본다
4. 사용자는 바로 공유 리스트 홈으로 진입한다

### 일상 사용
1. 사용자가 앱을 연다
2. 사용자는 즉시 미완료 항목을 본다
3. 사용자는 새 항목을 빠르게 추가하거나 기존 항목을 완료한다
4. 필요하면 항목을 열어 제목, 메모, 날짜를 수정한다
5. 완료된 항목은 별도의 보관 영역에서 확인한다

## 화면 구조
### 1. 진입 / 초대 흐름
목적: 최소한의 가입 마찰로 두 사람이 같은 공유 리스트에 들어오게 한다.

요구사항:
- 초대 링크 기반 진입 지원
- 계정 생성은 최대한 간단하게 유지
- 긴 온보딩 금지
- 가능한 적은 단계로 사용 가능한 리스트 상태에 도달

### 2. 홈 화면
목적: 지금 해야 할 항목을 가장 먼저 보여준다.

요구사항:
- 미완료 항목이 화면의 중심이어야 함
- 기본 정렬은 미완료 우선, 그 안에서는 최신 생성순
- 시각적 장식과 불필요한 화면 요소 최소화
- 새 항목 추가 동선이 명확해야 함
- 완료 항목이 메인 화면을 지배하면 안 됨

### 3. 항목 추가 / 수정 인터페이스
목적: 항목 생성과 수정이 거의 즉시 이루어지는 느낌을 만든다.

요구사항:
- 필드: 제목, 메모, 날짜
- 날짜 기본값은 생성일
- 메모에는 외부 링크 붙여넣기 가능
- 입력은 가벼운 시트, 인라인 편집, 또는 그에 준하는 저마찰 방식으로 제공

### 4. 완료 보관 영역
목적: 메인 리스트를 어지럽히지 않으면서 함께 쌓은 기록을 보존한다.

요구사항:
- 완료 항목은 미완료 항목과 분리
- 각 완료 항목에 완료일 표시
- 사용자는 항목을 다시 열 수 있음

## 상호작용 설계
- 홈 리스트에서 한 번의 탭으로 완료 처리
- 항목 행을 탭하면 수정 가능
- 매우 빠르고 즉각적으로 느껴지는 상호작용
- 이해를 돕는 수준의 절제된 모션만 사용
- 행동을 늦추는 장식용 애니메이션 금지

## 비주얼 디자인 방향
### 톤
- 텍스트 중심의 높은 미니멀함
- 장난스러운 감성보다 Apple 같은 절제된 정돈감
- 깔끔한 여백과 안정적인 세로 리듬
- 차분한 중성 톤과 높은 가독성

### UI 스타일
- 대부분 흰색 또는 아주 옅은 회색 배경
- 무거운 카드보다 얇은 구분선 중심
- 그림자는 최소화
- 명확한 타이포 위계
- 촘촘하지만 답답하지 않은 간격
- 터치 반응성과 가독성을 가장 우선시

### 피해야 할 것
- 과하게 감성적인 스티커나 일러스트
- 무거운 그라디언트
- 글래스모피즘
- 과도한 카드 스택
- 대시보드 같은 복잡한 구조

## 성능 목표
- 모바일 웹에서 빠른 첫 화면 표시
- 매우 낮은 JavaScript 용량
- 리스트 액션 시 최소한의 리렌더링
- 부드러운 스크롤과 리스트 갱신
- 명확한 이유 없이는 무거운 라이브러리 사용 금지

## 기술 방향
### 권장 플랫폼
가벼운 PWA 스타일 웹앱을 권장한다.

이유:
- 시도와 공유의 진입 장벽이 가장 낮다
- 높은 성능과 간단한 배포 구조를 만들기 쉽다
- 미니멀한 리스트 중심 제품과 잘 맞는다
- MVP 단계에서 네이티브 앱의 부담을 피할 수 있다

### 권장 아키텍처 방향
- 서버 기반 공유 상태를 사용하는 얇은 프론트엔드
- 클라이언트 상태 복잡성 최소화
- 단순한 인증 또는 준인증 수준의 초대 흐름
- 공유 리스트의 최신성 유지에 필요한 수준 이상의 실시간 복잡성은 배제

기능이 많은 협업 스택보다 단순한 공유 데이터 동기화 모델을 우선한다.

## 정렬 및 데이터 동작
### 홈 기본 정렬
1. 미완료 항목 우선
2. 미완료 항목 안에서는 최신 생성순
3. 완료 항목은 메인 활성 리스트에서 숨기고 보관 영역에서 보여줌

### 완료 처리 동작
- 항목을 완료하면 `completedAt`이 자동 저장된다
- 항목을 다시 열면 `completedAt`이 제거된다
- 완료일은 보관 영역에서 보여줘야 한다

### 날짜 동작
- 새 항목 날짜 기본값은 생성일이다
- 사용자는 필요하면 날짜를 수정할 수 있다
- 날짜는 앱의 주 정렬 기준이 아니라 보조 메타데이터다

## 오류 처리
앱은 단순해야 하며 기술적 복잡성을 사용자에게 드러내지 않아야 한다.

요구사항:
- 저장 실패 시 짧고 명확한 재시도 메시지 표시
- 잘못되었거나 만료된 초대 링크는 부드럽게 실패 처리
- 빈 상태에서는 첫 항목 추가를 자연스럽게 안내
- 오류 문구는 짧고 차분해야 함

## 테스트 우선순위
### 제품 수준 검증
- 새 사용자가 빠르게 리스트를 만들 수 있다
- 두 번째 사용자가 초대 링크로 혼란 없이 들어올 수 있다
- 사용자가 몇 초 안에 항목을 추가할 수 있다
- 사용자가 항목을 완료하고 다시 열 수 있다
- 완료일이 올바르게 저장되고 표시된다
- 홈은 항상 행동 가능한 미완료 항목을 우선한다

### 기술 수준 검증
- 모바일 첫 로드 성능이 충분히 좋다
- 현실적인 항목 수에서도 리스트 상호작용이 빠르다
- 불필요한 번들 비대화가 없다
- 메모 안의 외부 링크가 올바르게 열린다

## MVP 성공 기준
다음 조건을 만족하면 MVP는 성공이다.
- 두 사람이 매우 적은 설정만으로 하나의 리스트를 공유할 수 있다
- 일반적인 공유 메모 앱보다 더 빠르고 더 정돈된 느낌을 준다
- 추가 UI 개념을 학습하지 않고도 공유 목표를 관리할 수 있다
- 제품이 범용 소셜 앱으로 확장되지 않고 핵심 목적에 집중한다

## 이후 확장 아이디어
다음 항목은 MVP 범위 밖이지만, 제품이 유효하다고 판단되면 이후 확장 가능하다.
- 가벼운 리마인더
- 완료 항목 회고
- 완료 항목 사진 첨부
- 여러 개의 공유 리스트
- 약간의 테마 또는 리스트 개성화

## 권장 개발 철학
공유 리스트 경험을 매우 잘 전달하는 가장 작은 버전을 만든다. 화면 수, 상태 수, 선택지를 줄이는 것을 우선한다. 어떤 기능이 공유 목표 관리를 더 빠르게, 더 명확하게, 더 기분 좋게 만들지 못한다면 MVP에서는 제외한다.
