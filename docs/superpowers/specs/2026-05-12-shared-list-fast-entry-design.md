# 공유 리스트 빠른 연속 입력 설계

## 개요
공유 리스트 화면에서 사용자가 항목을 연속으로 빠르게 추가할 수 있도록 입력 흐름을 다듬는다. 목표는 화면 진입 직후 바로 입력할 수 있고, Enter만 반복해도 항목이 즉시 추가되며, 커서가 계속 같은 입력창에 남아 끊김 없이 다음 항목을 입력할 수 있게 만드는 것이다.

## 범위
### 포함
- 공유 리스트 화면(`src/components/local/shared-list-page.tsx`)의 새 항목 입력 경험 개선
- 첫 진입 시 새 항목 입력창 기본 포커스
- Enter 제출과 `+` 버튼 제출 후 입력값 초기화 및 포커스 유지
- 새 항목의 낙관적 추가
- 서버 저장 성공 시 임시 항목을 실제 항목으로 교체
- 서버 저장 실패 시 임시 항목 롤백
- 초기 포커스, Enter 등록, 포커스 유지, 연속 입력 테스트 보강

### 제외
- 홈 화면의 리스트 생성 입력 UX 변경
- 항목 상세 입력, 삭제, 공유, 정렬 등 다른 상호작용 변경
- 오프라인 저장, 재시도 큐, 전역 상태 도입

## 제품 원칙
- 첫 입력까지의 마찰이 없어야 한다.
- 연속 입력 흐름은 키보드만으로 끝나야 한다.
- 현재 구조를 유지하면서 변경 범위를 작게 가져간다.
- 저장 실패는 조용히 무시하지 않고 UI 상태를 원복해 일관성을 지킨다.

## 사용자 흐름
1. 사용자가 공유 리스트 화면에 처음 들어온다.
2. 새 항목 입력창이 기본 포커스를 가진다.
3. 사용자가 제목을 입력하고 Enter를 누르거나 `+` 버튼을 누른다.
4. 입력값은 즉시 목록에 새 미완료 항목으로 반영된다.
5. 입력창 값은 비워지고 포커스는 같은 입력창에 유지된다.
6. 사용자는 곧바로 다음 항목을 입력할 수 있다.
7. 서버 저장이 성공하면 임시 항목은 실제 서버 항목으로 대체된다.
8. 서버 저장이 실패하면 임시 항목은 목록에서 제거되고 입력창 포커스는 유지된다.

## 설계
### 1. 입력 포커스 관리
`shared-list-page`의 새 항목 input에 ref를 둔다. 첫 렌더 직후 자동 포커스를 보장하고, 항목 제출이 끝난 뒤에도 같은 ref에 다시 포커스를 준다. 브라우저 기본 동작에만 의존하지 않고 제출 후 명시적으로 `focus()`를 호출해 연속 입력 흐름을 고정한다.

### 2. 낙관적 항목 추가
제출 시점에 trim된 제목으로 임시 항목을 만든 뒤 `items` 상태의 미완료 목록 앞쪽에 즉시 삽입한다. 임시 항목은 클라이언트 전용 ID와 `completed_at: null`을 가지며, 렌더링에서 일반 항목과 동일하게 보인다. 이 변경은 사용자가 네트워크 지연 없이 즉시 등록되었다고 느끼게 한다.

### 3. 서버 동기화
낙관적 추가 직후 `addSharedListItem`을 호출한다. 성공하면 반환된 실제 항목으로 임시 항목을 교체한다. 실패하면 해당 임시 항목만 제거한다. 전체 `refresh()`를 항상 다시 호출하는 방식은 포커스 흔들림과 목록 점프를 만들 수 있으므로, 생성 직후에는 항목 단건 치환을 우선 사용한다.

### 4. 제출 중복 방지
같은 입력값으로 중복 제출되는 상황을 막기 위해 제출 중 상태를 둔다. 저장이 끝나기 전에는 추가 버튼을 비활성화하고, Enter 연타도 새 요청을 만들지 않게 한다. 대신 포커스는 유지되므로 저장이 끝나는 즉시 다음 입력을 이어갈 수 있다.

### 5. 목록 정렬 유지
임시 항목과 실제 항목 모두 현재 미완료 섹션에 유지한다. 생성 직후의 새 항목은 가장 위에 나타나야 한다. 완료 항목 정렬과 토글 동작은 이번 변경 범위 밖이므로 기존 동작을 그대로 둔다.

## 테스트
- 컴포넌트 테스트: 첫 렌더 후 새 항목 input이 포커스를 갖는다.
- 컴포넌트 테스트: 제목 입력 후 Enter 제출 시 입력값이 비워지고 포커스가 유지된다.
- 컴포넌트 테스트: 저장 성공 전에도 새 항목이 즉시 보인다.
- 컴포넌트 테스트: 저장 실패 시 임시 항목이 제거된다.
- E2E 또는 통합 테스트: 두 개 이상의 항목을 Enter만으로 연속 입력할 수 있다.

## 리스크와 대응
- 낙관적 항목과 실제 항목 교체 타이밍이 어긋나면 중복 항목처럼 보일 수 있다. 임시 ID를 기준으로 정확히 한 항목만 교체한다.
- 제출 후 전체 새로고침을 유지하면 포커스가 흔들릴 수 있다. 생성 경로에서는 전체 새로고침 대신 상태 치환을 사용한다.
- 저장 실패를 사용자에게 완전히 숨기면 항목이 사라지는 이유가 불명확할 수 있다. 이번 범위에서는 최소한 상태 롤백을 보장하고, 추가 오류 표시가 필요하면 다음 작업으로 분리한다.
